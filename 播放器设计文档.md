# ShortTV 播放器设计文档

> 目标：重新设计一个独立的 Android 播放器库 `player`，底层与当前项目保持一致，统一使用火山播放器 `TTVideoEngine`，并围绕短剧场景把播放器封装、安全、优化三部分做成模块化能力。

## 一、设计目标

- **目标 1：封装稳定**
  - 业务层只依赖统一的播放器接口，不直接操作底层播放器实现。
  - 沉浸页、Shorts 流、广告视频等场景共用同一套播放器内核。
- **目标 2：安全可控**
  - 支持播放地址校验、请求签名、URL 参数鉴权和火山播放器边播边解密。
  - 安全能力不散落在业务代码中，而是统一沉淀到库内部。
- **目标 3：优化可演进**
  - 首帧统计、断点续播、预加载、连续播放等能力统一放到优化层。
  - 后续更换播放器内核时，业务层和大部分优化/安全逻辑都可以复用。

## 二、模块结构

### 1）模块位置

- 新增 Android Library Module：`player`
- Gradle 接入：`settings.gradle` 新增 `include ':player'`

### 2）包结构

- `com.shorttv.player`
  - 对外接口、配置模型、播放器实现、播放器 view
- `com.shorttv.player.security`
  - 安全策略、请求签名、StrategySource 构建、解密回调
- `com.shorttv.player.optimize`
  - 首帧统计、断点续播、连续播放、预加载

## 三、播放器封装

### 1）设计思路

- **统一接口**
  - 对外只暴露 `ShortTvPlayer`，业务只需要关心 attach、setPlaylist、play、pause、resume、seekTo、release。
- **统一配置**
  - 通过 `PlayerConfig` 统一管理自动播放、循环、安全策略、优化策略。
- **统一视图**
  - 通过 `ShortTvPlayerView` 管理 `TextureView + Surface`，并把 renderView 交给 `TTVideoEngine` 做显示模式控制。

### 2）核心实现

- `ShortTvPlayer.kt`
  - 定义播放器接口、`PlayerItem`、配置模型和工厂，播放项支持 `playUrl` 和 `StrategySource` 两种火山播放器输入方式。
- `ShortTvPlayerView.kt`
  - 负责渲染 view 和 surface 生命周期管理。
- `DefaultShortTvPlayer.kt`
  - 使用 `TTVideoEngine` 作为统一内核，统一处理 strategySource 下发、回调接管、自动续播、首帧、进度保存和自动播下一集。

### 3）接口与数据模型细节

- `ShortTvPlayer` 能力边界
  - `attach(playerView)`：仅负责绑定渲染容器与 surface 回调，不触发播放。
  - `setPlaylist(items, startIndex)`：写入播放列表；当 `PlayerConfig.autoPlay=true` 时会立即触发 `play(startIndex)`。
  - `play(index)`：以“播放列表 + 当前索引”为中心驱动播放，不暴露底层 engine 的复杂状态机。
  - `seekTo(positionMs)`：仅在 `prepared=true` 后生效，seek 生命周期通过事件/状态回调对外可观测。
  - `release()`：释放 engine、解绑 surface、清理运行时状态，确保不会残留渲染与线程资源。
- `PlayerItem` 输入模式
  - `playUrl`：直接播放单一路径的 URL，适合已经在业务端完成编排、加签后的直链。
  - `strategySource`：交给火山策略源，适合多清晰度/复杂编排/配合预加载或后续 ABR 的场景。
  - `fallbackSources`：用于“源级别切换”（例如不同清晰度直链/不同线路/不同 CDN），由优化层在卡顿或错误时切换并重播。
  - `cacheKey`：用于缓存/预加载的稳定 key，避免 URL 带签名参数导致缓存命中下降。
- 播放源的语义拆分
  - “引擎内部切清晰度”（同一个 source 内切轨）与“切 fallbackSources”（换 source 重播）是两层能力，面向业务要分别暴露“当前实际清晰度”与“当前使用的 source 档位”。

### 4）生命周期与回调模型

- 生命周期主流程
  - `setPlaylist()` → `play()`：进入 `PREPARING`，准备下发播放源与创建 engine。
  - `onPrepared()`：进入 `PREPARED`，表示底层资源可用。
  - `onRenderStart()`：进入 `PLAYING`，同时产生首帧事件（用于体验指标）。
  - `onCompletion()`：进入 `COMPLETED` 或自动播下一集（取决于 `loopCurrentItem` 与列表是否还有下一集）。
- 事件 vs 状态
  - 状态：表达“现在能做什么”，需要稳定持续（例如 `PLAYING/PAUSED/SEEKING`）。
  - 事件：表达“刚刚发生了什么”，瞬时通知（例如 `FirstFrameRendered/BufferingStart/SeekCompleted/Retry/QualityDowngraded`）。
  - 这套拆分的目标是避免业务侧用一堆零散回调拼状态，提升 UI 和埋点一致性。

### 5）封装价值

- 页面不再直接依赖播放器 SDK
- 场景复用更容易
- 后续切换到底层新内核时改造成本更低

## 四、安全

### 1）设计思路

- **远端播放安全**
  - 播放前统一校验地址协议，阻断不安全的 http 裸链。
- **链路安全**
  - 围绕火山播放器场景，统一给播放 URL 追加 token、nonce、requestId、signature 等签名参数。
- **加密播放安全**
  - 对接火山播放器的 `BufferProcessCallback`，支持类似当前项目的边播边解密能力。

### 2）核心实现

- `DefaultPlaybackSecurityManager`
  - 负责把 `PlayerItem` 转换成安全可播放的 `StrategySource`。
- `RequestSigner`
  - 抽象 URL 签名能力，默认实现使用简单签名规则，方便后续替换成真实业务算法。
- `SecuredPlaybackSource`
  - 统一承载签名后的 `StrategySource` 和解密开关。
- `BufferProcessCallback + PlayerDecryptor`
  - 对齐当前项目的火山播放器解密方案，在播放器读取数据时执行分片解密。

### 3）安全处理链路（从输入到可播放 source）

- 输入：业务层只提供 `PlayerItem`（可能是 `playUrl`，也可能是 `strategySource`）。
- 校验：在真正创建/下发 `strategySource` 前完成协议校验（例如 require https）。
- 补参与签名
  - 如果配置了 token/nonce provider：在 URL 上追加 query 参数（例如 `token=xxx&nonce=xxx`）。
  - `RequestSigner` 负责把“当前 URL + 业务标识”转换成签名字段（例如 requestId、signature），并统一追加。
  - 目标是把“播放请求生成规则”固定在播放器库内部，避免各页面散落拼参导致线上不可控。
- 输出：统一产出 `SecuredPlaybackSource`
  - `strategySource`：给 engine 使用的可播放输入（直链也会被规整到策略源/直链源的统一结构上）。
  - `needDecrypt`：是否需要挂载解密回调。

### 4）扩展点与风险边界

- 播放协议与域名策略
  - `PlaybackSecurityPolicy.requireHttps` 只负责“协议合规”；更细粒度的 host allowlist/黑名单可以在 security 层扩展实现，保证策略收口。
- 鉴权的职责划分
  - 播放器库负责“生成合规的请求参数与签名字段”并随播放请求发送。
  - 真正的鉴权逻辑（校验 token/签名/权限）由业务服务或 CDN 边缘完成，播放器不做“回源鉴权判定”。
- 加解密职责边界
  - 加密在服务端产生，播放器通过 `BufferProcessCallback` 在读取数据时分片解密，实现“边下边播边解密”。
  - 解密 key 的来源属于业务安全体系的一部分，这里通过 `PlaybackSecurityPolicy` 统一注入，避免页面散落。

### 5）安全价值

- 避免页面层拼接不一致的播放请求
- 把内容安全和链路安全统一沉淀在播放器库内
- 把加密播放能力和火山播放器内核绑定在库内部

## 五、优化

### 1）首帧优化

- `FirstFrameTracker`
  - 在发起播放请求时记录时间，在首帧出现时计算首帧耗时。
- `DefaultPlaybackOptimizationManager`
  - 统一管理首帧指标，并在播放器创建时接入 ABR 起播策略。
- 价值
  - 便于对短剧首帧体验做量化和持续调优

### 2）连续播放优化

- `PlaylistCoordinator`
  - 负责判断下一集是谁，以及当前集准备完成后要预热哪些后续资源。
- `PlaybackPositionStore`
  - 负责保存和恢复断点续播位置。
- `DefaultShortTvPlayer`
  - 在播放结束时自动切下一集，在短剧场景中形成连续播放链路。

### 3）预加载与稳定性优化

- `UrlPreloadManager`
  - 基于火山播放器预加载能力，通过 `TTVideoEngine.addTask` 对下一条视频做预热。
- `PlaybackOptimizationPolicy`
  - 支持配置 preloadWindow、preloadSizeBytes、首帧告警阈值、是否启用 ABR 起播和引擎预加载。
- 价值
  - 降低下一集起播延迟
  - 降低滑动切换时的等待感
  - 提高连续消费体验

### 4）卡顿、重试与自动降档

- 卡顿观测（buffer tracker）
  - 记录每个 item 的 buffering 次数、开始时间、结束耗时、是否发生在首帧后、buffer 原因与 action。
  - 目标是让“卡顿”不仅可见，还能作为后续策略输入：什么时候降档、是否需要告警、是否需要上报。
- 起播重试（startup retry）
  - 只针对“首帧前错误”做有限次数重试，避免短剧用户看到“秒失败”。
  - 设计上与“自动降档”解耦：先提高成功率，再考虑体验降级。
- 自动降档（quality downgrade）
  - 通过 `fallbackSources` 维护“源级别档位”，当卡顿次数/时长达到阈值或遇到特定错误码时切到更低档位并重播。
  - 该能力用于兜底弱网与错误恢复，不依赖引擎 ABR 是否开启。

### 5）ABR 起播配置（与降档的关系）

- 起播 ABR 的目的
  - 短剧场景更关心“首帧速度”，因此会通过 gear strategy 等能力让引擎在起播阶段选择更稳妥的初始档位，降低首帧风险。
- 与自动降档的关系
  - ABR：同一个 source 内的动态切轨（更平滑，通常不断流）。
  - 自动降档：切换 `fallbackSources`（换 source 重播，兜底更强但更重）。
  - 两者可以共存：ABR 解决“可切轨时的体验”，自动降档解决“源不可用/持续卡顿/错误恢复”的兜底。

### 6）播放优化注意点

- **首帧优先**
  - 短剧场景下最关键的是从 `play()` 到 `onRenderStart` 的耗时，必须持续统计首帧成本并设置阈值告警。
- **连续播放优先**
  - 优化目标不能只看单视频起播，还要看切下一集是否顺滑，因此要把自动播下一集和预加载一起设计。
- **预加载适度**
  - 一般只预热后续 1 集最稳，窗口过大会放大流量、内存和缓存压力。
- **缓存 Key 稳定**
  - 播放 URL 往往会带 token、nonce、signature 等动态参数，缓存 key 不能直接跟着签名 URL 变化，应该优先使用稳定的业务标识。
- **卡顿可观测**
  - 不仅要知道有没有卡顿，还要知道是第几次卡顿、卡顿持续多久、发生在首帧前还是首帧后。
- **错误可恢复**
  - 对首帧前异常要预留重试空间，避免用户感知为直接播放失败。
- **弱网可降档**
  - 原项目里已经有播放报错和网络卡顿时自动切到更低画质的策略，组件化后也应该保留，否则稳定性能力会倒退。
- **状态要统一**
  - 除了分散的回调事件，播放器还应该提供统一的播放状态流转，比如 `IDLE`、`PREPARING`、`PREPARED`、`PLAYING`、`PAUSED`、`SEEKING`、`COMPLETED`、`ERROR`，这样业务层更容易做 UI 控制和操作合法性控制。
  - 状态只表达“现在能做什么”，因此必须稳定、持续一段时间。
  - `first frame rendered`、`buffering start/end`、`seek completed`、`retry`、`quality downgrade` 都属于事件，不应该放进状态枚举里。

## 七、为什么这个方案适合短剧

- **封装上**
  - 业务层不用理解底层播放器复杂细节，可以更专注在剧集编排和交互。
- **安全上**
  - 短剧内容资产更敏感，必须考虑加密播放和播放链路保护。
- **优化上**
  - 短剧用户最敏感的是首帧、切集、连续播放和稳定性，这个库围绕首帧、预加载、稳定缓存 key、卡顿观测、起播重试、弱网自动降档和统一播放状态做了针对性设计，并且把“状态”和“事件”明确拆开，避免状态机语义混乱。
